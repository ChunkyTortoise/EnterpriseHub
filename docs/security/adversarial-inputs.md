# Adversarial Inputs

This note describes how EnterpriseHub handles hostile or malformed input. It replaces a full STRIDE document on purpose (see REQ-W4-1): the threat surface here is narrow (signed CRM webhooks plus free-text chat turns), so a short grounded write-up is more useful than a generic threat matrix.

At the perimeter, GoHighLevel webhooks are authenticated before any handler runs. `ghl_integration/validators.py` computes an HMAC-SHA256 digest over the raw payload using `GHL_WEBHOOK_SECRET` and compares it with `hmac.compare_digest` (constant-time, so signature checks do not leak timing); unsigned requests are rejected, and enterprise tenants can use an RSA scheme instead. Paid-API callers go through a parallel fail-closed signing path in `platform/api_monetization.py`, which raises at startup if the signing secret is missing or under 32 characters (REQ-W4-2). Free-text reaching the bots is screened by `security/input_validator.py`, whose rule set rejects or sanitizes SQL, XSS, command, LDAP, XXE, and NoSQL injection patterns; prompt-style content is further bounded by truncating each turn to a maximum length before model processing. Abuse volume is capped at two layers: `security/rate_limiter.py` enforces Redis-backed limits across IP, user, API-key, and endpoint scopes (sliding-window, token-bucket, and adaptive strategies), and `utils/conversation_rate_limiter.py` adds a per-contact ceiling of 30 messages per hour and 200 per day to stop a single lead from flooding a bot.

Every security-relevant action is recorded for forensics. `security/audit_logger.py` emits typed `AuditEventType` records (login attempts, PII access, rate-limit hits) with hash-linked entries and Prometheus counters (`security_events_total`, `auth_failures_total`, `pii_access_total`), all flowing through the structlog JSON-per-line pipeline configured in `logging_config.py` so events are queryable in Loki or Elasticsearch. Compliance edge cases are handled where they matter most, in the conversation path rather than as a single gate. TCPA opt-out is the strictest: `agents/jorge_buyer_bot.py` checks opt-out phrases before any AI processing and tags the contact `AI-Off`, backed by the immutable `sms_opt_outs` and `sms_consent_tracking` tables from migration `005_sms_compliance_tables.sql`. Fair Housing protection is a work in progress: `compliance_platform/engine/policy_enforcer.intercept_message` exposes pattern-based detection of protected-class and steering language (policies FHA-001/FHA-002), but it is not yet wired into the live reply path, so today it is a capability the engine offers rather than an enforced pre-send check. RESPA is treated as a design constraint (no steering to affiliated lenders) surfaced through the `respa_compliance` tenant flag in `white_label/tenant_manager.py` and the FHA/RESPA hook in `agents/voss_negotiation_agent.py`, which is currently a stub. These three are called out explicitly because they are the regulatory inputs most likely to be tested adversarially, and because the gap between guarded and fully enforced is the honest current state.
